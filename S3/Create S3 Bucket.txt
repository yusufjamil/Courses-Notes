What is an S3 Bucket?

An Amazon S3 Bucket is a logical container used to store objects (files) in Amazon S3.

Every file uploaded to Amazon S3 must reside inside a bucket.

Examples of objects stored in a bucket include:

Images
Videos
PDF documents
ZIP archives
Application files
Log files
Backup files

Choose AWS Region

Every bucket is created in an AWS Region.

Examples:

Asia Pacific (Mumbai)
Asia Pacific (Singapore)
US East (N. Virginia)
Europe (Frankfurt)

Why the Region Matters

The selected Region determines:

Where your data is stored.
Data residency and compliance.
Latency for users.
Potential data transfer costs.

Best Practice: Choose the Region closest to your users or where your application runs.

Bucket Name

The bucket name must be entered during creation.

Bucket Naming Rules
Must be globally unique across AWS.
Use only lowercase letters.
Numbers are allowed.
Hyphens (-) are allowed.
No spaces.
No uppercase letters.
No special characters other than hyphens (and periods in some cases, though they are generally avoided).


Examples

Valid

company-backups
student-project-files
aws-training-demo

Invalid

MyBucket
AWS Bucket
Backup_Files

Object Ownership

Makes the bucket owner the owner of all uploaded objects.
Disables Access Control Lists (ACLs).
Simplifies permission management.

For most new projects and AWS best practices, this is the preferred configuration.

Block Public Access

Purpose

It helps prevent accidental exposure of bucket contents to the internet.

When enabled:

Public bucket policies are blocked.
Public ACLs are ignored.
Anonymous users cannot access bucket contents.

For most applications, these settings should remain enabled.

Only disable them when you intentionally need a public bucket (for example, certain static website hosting scenarios).

Bucket Versioning

What is Versioning?

Versioning allows Amazon S3 to keep multiple versions of the same object.

Example

Upload:

report.pdf

Later, upload another file with the same name.

Without versioning:

The original file is overwritten.

With versioning:

Both versions are preserved.
Older versions can be restored if needed.

Benefits
Protects against accidental deletion.
Protects against accidental overwrite.
Simplifies recovery of previous file versions.

For production environments, versioning is commonly enabled.

Review Configuration

Before creating the bucket, review:

Bucket name
Selected Region
Object Ownership
Public access settings
Versioning

Bucket Lifecycle After Creation

Once created, you can:

Upload files.
Create folders (prefixes).
Download objects.
Delete objects.
Configure permissions.
Enable encryption.
Apply lifecycle rules.
Configure replication.
Host static websites (if required).

Common Beginner Mistakes
Choosing a bucket name that is already in use.
Selecting the wrong AWS Region.
Disabling Block Public Access unnecessarily.
Forgetting to enable Versioning for important data.
Using unclear bucket names that make management difficult.

Best Practices
Use descriptive, globally unique bucket names.
Keep Block Public Access enabled unless public access is explicitly required.
Enable Versioning for important or production data.
Use IAM policies instead of broad public permissions.
Enable server-side encryption for sensitive data.
Organize objects with logical prefixes (for example, images/, backups/, logs/).

Real-World Example

A company stores customer invoices in an S3 bucket.

Bucket name: company-invoices
Region: Asia Pacific (Mumbai)
Versioning: Enabled
Block Public Access: Enabled

If an employee accidentally uploads an incorrect invoice over an existing one, versioning allows the previous version to be restored.


What is an Object?

Everything stored inside an Amazon S3 bucket is called an Object.

Examples of objects include:

PDF files
Images
Videos
ZIP files
Excel sheets
Word documents
Text files
Backup files
Log files

Unlike a traditional file system, Amazon S3 stores data as objects, each containing:

The actual file data
Metadata
A unique object key

Amazon S3 allows uploading:

A single file
Multiple files
Entire folders (through supported interfaces or tools)

This is useful when migrating application assets, backups, or datasets.

Object Key

Every uploaded object receives a unique Object Key.

Example:

Bucket:

student-records

Objects:

student-records
├── student1.pdf
├── student2.pdf
├── photo.jpg
└── resume.docx

Here:

student1.pdf
student2.pdf
photo.jpg
resume.docx

are object keys.

Object URL

Each object has a URL that can be used to access it.

However, having a URL does not mean the object is publicly accessible.

Access depends on:

IAM permissions
Bucket policy
Object ACLs (if enabled)
Block Public Access settings
Metadata

Metadata is additional information associated with an object.

Examples include:

Content type (e.g., image/jpeg, application/pdf)
Content length
Last modified time
Custom metadata added by applications

Metadata helps applications identify and process files correctly.

Storage Class

Every object belongs to a storage class.

Common storage classes include:

S3 Standard
S3 Intelligent-Tiering
S3 Standard-IA
S3 One Zone-IA
S3 Glacier Instant Retrieval
S3 Glacier Flexible Retrieval
S3 Glacier Deep Archive

By default, uploaded objects use S3 Standard unless another class is selected.

If Versioning is enabled, AWS creates a delete marker instead of permanently removing the object immediately.

Creating Folders

Although Amazon S3 is an object storage service and does not use real directories, the console allows users to create folders.

Internally, folders are represented using prefixes.

Example:

company-data
│
├── Images/
│      logo.png
│      banner.jpg
│
├── Documents/
│      invoice.pdf
│      contract.docx
│
└── Backups/
       backup.zip

Folders improve organization and make objects easier to locate.

Common Bucket Operations

Inside a bucket, you can:

Upload objects
Download objects
Delete objects
Rename objects (typically by copying and deleting)
Create folders (prefixes)
Copy objects
Move objects
View properties
Change permissions
Enable encryption
Configure lifecycle policies


Real-World Example

A software company stores application assets in Amazon S3:

Bucket:

company-assets

Folders:

Images/
Videos/
Documents/
Backups/

Developers upload:

Company logos
User manuals
Training videos
Backup archives

Applications retrieve these files directly from Amazon S3 when needed.

Best Practices
Use meaningful object names.
Organize data with prefixes (folders).
Avoid storing unrelated data in the same bucket without structure.
Use encryption for sensitive files.
Enable Versioning for important documents.
Regularly delete unused objects to reduce storage costs.
Apply least-privilege IAM permissions.

Common Beginner Mistakes
Uploading files to the wrong bucket.
Accidentally deleting important objects.
Assuming object URLs are automatically public.
Not organizing files with folders/prefixes.
Ignoring storage class selection for long-term data.
